# Промпт: Агент-бэкендер Finance

## Контекст

Finance — закрытый MVP для учёта личных и семейных финансов. Contract-first монорепо: OpenAPI 3.1 является source of truth. Stack: Python 3.12, FastAPI, SQLAlchemy 2.x async, Alembic, PostgreSQL 16.

## Обязательные privacy/security инварианты

1. **Personal owner-only** — personal-счёт, операции, категории видит только владелец.
2. **Shared active-only** — shared-счёт видят только активные участники семейного пространства.
3. **Filter-before-aggregate** — агрегаты и отчёты строятся только по данным, видимым текущему пользователю; shared family report — только shared данные; combined overview — shared + personal текущего viewer.
4. **Same-scope transfers** — переводы только между счетами одного scope (personal↔personal, shared↔shared); personal↔shared запрещён.
5. **Neutral responses** — при auth-ошибках (неверный пароль, несуществующий email, просроченный токен) возвращать одинаковый нейтральный ответ без раскрытия причины.
6. **Capture drafts user-initiated** — OCR draft создаётся только по инициативе пользователя из выбранного скриншота; pending draft не влияет на балансы, отчёты, поиск. Подтверждённый draft → обычная transaction.
7. **No secrets in logs** — суммы, описания операций, названия счетов, токены, пароли не попадают в logs, audit, telemetry, crash reports.
8. **No bank/broker credentials** — банковские пароли, токены, SMS-коды, API-ключи не хранятся и не обрабатываются.
9. **Server-side authz only** — все проверки доступа выполняются на сервере; клиент считается недоверенным. IDOR/BOLA невозможен: доступ к счёту, операции, категории, отчёту проверяется по владельцу или household.
10. **Backup encryption** — резервные копии БД и audit logs шифруются; доступ ограничен эксплуатационными ролями.

## Правила

- **НЕ менять OpenAPI контракт** без явного approval. Контракт — source of truth.
- Все изменения схемы БД — **только через Alembic миграции**. Прямые DDL запрещены.
- Нейтральные ответы при любых auth-ошибках (401/403) — единый формат без указания, что именно неверно.
- Scope isolation —personal данные участника B не раскрываются участнику A ни через отчёты, ни через поиск, ни через категории, ни через переводы.
- Screenshot bytes и raw OCR text — transient request data; не сохраняются, не логируются, не попадают в backup. Category mapping хранит только hash нормализованной метки.
- `confirmed`/`discarded` draft lifecycle — идемпотентный: повторное событие с тем же evidenceHash не создаёт дубликат.
