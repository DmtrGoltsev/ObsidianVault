Правила Git-изоляции для агентной работы:

Полная инструкция по работе с GitHub: [[Регламент_GitHub_для_Агентов]]

1. Любые изменения кода, документации, Obsidian vault или конфигурации делай только в отдельной Git-ветке.
2. Никогда не изменяй напрямую ветки `main`, `master`, `develop`, `release/*`.
3. Никогда не выполняй `git merge`, `git rebase`, `git push`, `git push --force`, `git reset --hard` без моего явного отдельного разрешения.
4. Merge в основную ветку всегда выполняю только я вручную после проверки.

Перед началом работы выполни preflight:
- проверь, что это Git-репозиторий: `git status`;
- проверь текущую ветку: `git branch --show-current`;
- проверь незакоммиченные изменения: `git status --short`;
- если есть чужие незакоммиченные изменения, не трогай их и остановись с вопросом;
- если текущая ветка `main`, `master`, `develop` или `release/*`, создай новую ветку до любых изменений.

Именование веток:
- формат: `agent/<system>/<task-slug>`
- примеры:
  - `agent/opencode/task-tracker-plan`
  - `agent/deepseek/android-task-steps`
  - `agent/codex/obsidian-memory-rules`

Если параллельно работают несколько агентов:
- каждый агент должен работать в своей ветке;
- при возможности используй отдельный `git worktree`;
- не редактируй файлы, которые уже изменяет другой агент, если задача явно не про интеграцию.

Порядок работы:
1. Создай или переключись на отдельную ветку.
2. Сделай изменения только в рамках задачи.
3. Запусти доступные проверки/тесты.
4. Не мержи изменения.
5. В конце верни отчет:
   - имя ветки;
   - какие файлы изменены;
   - какие команды/тесты запускались;
   - результат проверок;
   - что осталось сделать;
   - есть ли конфликты или риски для merge.
