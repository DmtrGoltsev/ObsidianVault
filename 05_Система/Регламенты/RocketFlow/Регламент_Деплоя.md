# Регламент Деплоя

## Цель

Стандартизировать процедуру развёртывания RocketFlow на production-сервер [[HexCore]].

## Область применения

Деплой backend jar и web статики на HexCore (45.10.110.42). Docker/GHCR deploy не является закрытым production-фактом и остаётся open gate.

## Процедура

### Backend (systemd)

1. Собрать backend jar через CI (`backend-verify` workflow)
2. Остановить systemd unit: `sudo systemctl stop rocketflow-backend`
3. Скопировать jar на HexCore
4. Запустить unit: `sudo systemctl start rocketflow-backend`
5. Проверить статус: `sudo systemctl status rocketflow-backend`
6. Smoke check: `curl http://localhost:8080/actuator/health`

### Web (Nginx)

1. Собрать production build: `npm run build` (через `web-verify`)
2. Скопировать статику в Nginx docroot
3. Проверить доступность web через HTTPS

### Откат

- Восстановить предыдущую версию jar и перезапустить systemd unit

## Участники

- [[Агент_DevOps]] — исполнитель деплоя
- [[Агент_Бэкенд]] — предоставляет jar
- [[Агент_Веб]] — предоставляет статику
- [[Агент_QA]] — smoke-валидация после деплоя

## Триггеры запуска

- Ручной: workflow_dispatch (`backend-hexcore-prod-deploy.yml`)
- Автоматический: после merge в main (при настроенном CI)

## Критерии успеха

- `/actuator/health` возвращает UP
- Web доступен по HTTPS
- Основные endpoints отвечают 200

## Эскалация

Отказ деплоя → откат → диагностика логов (`journalctl -u rocketflow-backend`) → [[Агент_DevOps]] → [[Оркестратор]].
