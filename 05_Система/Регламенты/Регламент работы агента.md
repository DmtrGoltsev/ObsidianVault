# Регламент работы агента

Ты работаешь как агент разработки проекта с использованием Obsidian как проектной памяти.

## Язык

- Все папки, файлы заметок, заголовки, описания, комментарии и документация в Obsidian должны быть на русском языке.
- Английский допускается только для технических названий: API, SDK, CLI, Git, DeepSeek, OpenCode, npm, Android, Kotlin, Gradle, имена файлов кода, команды, endpoints, классы, функции, переменные и названия библиотек.

## Obsidian vs Git

Obsidian vault используется как память проекта, а не как замена Git.
- Код, diff, история изменений и реализации живут в Git.
- Obsidian хранит цель проекта, контекст, задачи, решения, источники, доказательства, промпты, регламенты и связи между ними.
- Правила работы с Git и GitHub: [[Регламент_GitHub_для_Агентов]]
- Правила изоляции веток: [[Регламент Git-изоляции агентов]]

## Перед началом задачи

1. Найди или создай заметку проекта в `01_Работа/Проекты/Проект/`.
2. Найди или создай заметку задачи в `01_Работа/Задачи/Проект/`.
3. Подготовь пакет контекста в `02_Знания/Пакеты_контекста/Проект/`.
4. Проверь связанные решения в `04_Решения/Проект/`.
5. Проверь источники и доказательства в `02_Знания/Источники/Проект/` и `02_Знания/Доказательства/Проект/`.

## После выполнения задачи

1. Обнови заметку задачи: статус, что сделано, какие файлы изменены, что осталось.
2. Добавь доказательства: тесты, команды, выводы, скриншоты, ссылки на commit/PR, если есть.
3. Если принято архитектурное или процессное решение, создай или обнови заметку в `04_Решения/Проект/`.
4. Если появились новые термины, обнови `02_Знания/Глоссарий/Проект/`.
5. Если контекст изменился, обнови пакет контекста.

## Правила качества

- Любой факт должен иметь ссылку на источник или доказательство.
- Если доказательства нет — пометь утверждение как предположение, `уверенность: низкая`.
- Не сохраняй секреты: API keys, токены, пароли, cookies, private keys, `.env`, персональные данные.
- Не копируй большие куски кода в Obsidian. Ссылайся на путь файла, branch, commit, функцию или класс.
- Не удаляй историю решений. Устаревшее помечай как `заменено` или `архив`.
- Используй Obsidian-ссылки формата `[[...]]`.

## Допустимые типы заметок

| Тип | Назначение | Папка |
|-----|-----------|-------|
| `проект` | Главная заметка проекта | `01_Работа/Проекты/Проект/` |
| `задача` | Активная задача | `01_Работа/Задачи/Проект/` |
| `агент` | Роль агента | `03_Агенты/Проект/` |
| `пакет_контекста` | Пакет контекста для агента | `02_Знания/Пакеты_контекста/Проект/` |
| `источник` | Ссылка на документ/файл | `02_Знания/Источники/Проект/` |
| `доказательство` | Результат верификации | `02_Знания/Доказательства/Проект/` |
| `решение` | ADR | `04_Решения/Проект/` |
| `термин` | Статья глоссария | `02_Знания/Глоссарий/Проект/` |
| `регламент` | Правило/процедура | `05_Система/Регламенты/` или `Проект/` |
| `промпт` | Промпт для агента | `05_Система/Промпты/Проект/` |
| `ревью` | Результат ревью | `01_Работа/Ревью/` |
| `MOC` | Навигационная карта | `05_Система/Карты_MOC/` или `Проект/` |
| `схема` | Схема БД/развертывания | `05_Система/Схемы/Проект/` |

## Допустимые статусы

| Статус | Значение |
|--------|----------|
| `черновик` | Заметка создана, содержание не финальное |
| `активно` | В работе |
| `на_ревью` | Ожидает проверки |
| `утверждено` | Проверено и принято |
| `заблокировано` | Зависит от внешнего фактора |
| `выполнено` | Задача завершена |
| `устарело` | Информация потеряла актуальность |
| `заменено` | Заменено другой заметкой |
| `архив` | Перемещено в `99_Архив/` |

## Формат ответа агента

В конце каждого ответа агент обязан вернуть:

1. Какие заметки нужно создать или обновить
2. Готовый Markdown для этих заметок
3. Какие связи `[[...]]` добавить
4. Какие доказательства есть
5. Какие доказательства отсутствуют
6. Риски
7. Уровень уверенности
