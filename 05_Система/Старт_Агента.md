# Старт агента — точка входа в базу знаний

Ты — агент. Ты находишься в Obsidian vault — базе знаний проектов. Этот документ объясняет, как здесь всё устроено, как ориентироваться, где искать информацию и по каким правилам работать.

Прочитай до конца. Это единственный документ, который тебе нужен для входа.

---

## 1. Как устроен vault

Vault — это иерархия папок с Markdown-заметками. Каждая папка имеет чёткое назначение.

```
00_Входящие/              ← новые нераспределённые задачи
01_Работа/
  Проекты/Проект/         ← главные заметки проектов: цель, стек, статус, волны
  Задачи/Проект/          ← активные задачи: критерии, блокеры, доказательства
  Ревью/                  ← результаты ревью
02_Знания/
  Глоссарий/Проект/       ← доменные и технические термины
  Источники/Проект/       ← ссылки на документы и ключевые файлы репозитория
  Доказательства/Проект/  ← доказательства верификации: тесты, CI, E2E
  Пакеты_контекста/Проект/ ← пакеты контекста для агентов
03_Агенты/Проект/         ← роли агентов: зоны ответственности, права решений
04_Решения/Проект/        ← архитектурные решения (ADR)
05_Система/
  Карты_MOC/MOC_Все_Проекты.md  ← хаб всех проектов (точка входа в навигацию)
  Карты_MOC/Проект/       ← MOC-карты: главная, бэкенд, веб, Android, DevOps
  Промпты/Проект/         ← промпты для агентов
  Регламенты/             ← общие регламенты (работа агента, GitHub, Git-изоляция)
  Регламенты/Проект/      ← проектные регламенты
  Схемы/Проект/           ← схемы БД, развертывания
  Шаблоны/                ← шаблоны для создания новых заметок
  Старт_Агента.md         ← этот документ
Активы/                   ← скриншоты, изображения
99_Архив/                 ← устаревшие заметки
```

**Принцип:** общие ресурсы лежат в корнях папок. Всё, что относится к конкретному проекту — в подпапке `Проект/`. Новый проект добавляется созданием таких же подпапок рядом.

## 2. Как ориентироваться

### Найти проект
1. Открой [[MOC_Все_Проекты]] — верхнеуровневый хаб всех проектов
2. Выбери проект → открой его MOC-карту (например `MOC_ИмяПроекта`)
3. Из MOC-карты иди глубже:
   - **Главная заметка проекта** — `01_Работа/Проекты/Проект/` — цель, стек, статус, активные гейты
   - **Под-MOC** — карты компонентов (бэкенд, фронтенд, мобилка, DevOps)
   - **Глоссарий** — `02_Знания/Глоссарий/Проект/` — термины, чтобы говорить на одном языке
   - **Решения** — `04_Решения/Проект/` — почему сделано так, а не иначе
   - **Пакеты контекста** — `02_Знания/Пакеты_контекста/Проект/` — готовая выжимка для старта
   - **Доказательства** — `02_Знания/Доказательства/Проект/` — что уже проверено
   - **Агенты** — `03_Агенты/Проект/` — кто за что отвечает в проекте

### Быстрый путь для новой задачи
1. Открой [[MOC_Все_Проекты]] → выбери проект → открой его MOC
2. Возьми пакет контекста: `02_Знания/Пакеты_контекста/Проект/`
3. Изучи глоссарий проекта
4. Проверь существующие решения
5. Создай заметку задачи в `01_Работа/Задачи/Проект/` по [[Шаблон_Задача]]
6. Работай по правилам ниже

### Граф
Граф Obsidian (`Ctrl+G`) покажет все связи визуально. Цвета и связи помогут быстро найти нужный кластер знаний.

## 3. Правила работы — обязательно

### Язык
- **Русский** — основной для всего: папок, файлов, заголовков, описаний, документации
- **Английский** — только для технических названий: API, SDK, CLI, Git, GitHub, npm, имена библиотек, фреймворков, классов, функций, файлов кода, endpoints, команд
- При добавлении нового контента: сначала русская версия

### YAML frontmatter
**Обязателен** для всех заметок, кроме шаблонов.

```yaml
---
id: ""                    # уникальный id заметки
тип: ""                   # проект|задача|агент|пакет_контекста|источник|доказательство|решение|ревью|промпт|термин|регламент
статус: ""                # черновик|активно|на_ревью|утверждено|заблокировано|выполнено|устарело|заменено|архив
проект: ""                # имя проекта или "" для общих заметок
владелец: ""
создано: ""
обновлено: ""
уверенность: ""           # высокая|средняя|низкая
источники: []             # ссылки на файлы-источники
доказательства: []        # ссылки на доказательства
теги: []
---
```

### Obsidian-ссылки
- Формат: `[[ИмяФайла]]` — без расширения `.md`
- Связывай заметки перекрёстно: каждая заметка должна ссылаться на связанные и иметь входящие ссылки
- MOC-карты — главный механизм навигации: они собирают ссылки на все дочерние заметки

### Код
- **НЕ копировать** большие куски кода в Obsidian
- Ссылаться на путь файла, строку, функцию, класс: `src/module/file.ts:42`
- Obsidian — память проекта, Git — хранилище кода

### Секреты — запрещены
Никогда не сохранять в vault:
- API keys, токены, пароли, cookies
- Private keys, `.env` файлы
- Персональные данные пользователей

### Доказательства
- Любой факт должен иметь ссылку на источник
- Нет источника → `уверенность: низкая` + пометка `(предположение)` в тексте
- После задачи: обнови статус, добавь доказательства (тесты, выводы команд, ссылки на commit)

### Git
- Всегда работай в отдельной ветке: `agent/<среда>/<задача>`
- Никогда не пушишь в `main`/`master` напрямую
- Полные правила: [[Регламент_GitHub_для_Агентов]] + [[Регламент Git-изоляции агентов]]
- Vault сам является Git-репозиторием — работай с ним по тем же правилам

### История
- Не удаляй устаревшие заметки — помечай `статус: устарело` или `заменено`
- Переноси в `99_Архив/` только когда заметка точно не нужна

## 4. Оркестрация (если ты часть мультиагентной системы)

### Ты — субагент
Если тебя запустил оркестратор:
- Ты выполняешь **только** выданную подзадачу
- Не запускаешь собственного планировщика (если не поручено явно)
- Следуешь контракту запуска: роль, уровень рассуждения, критерии, бюджет, стоп-условия
- Возвращаешь: сводку, артефакты, допущения, доказательства, блокеры, риски, статус по каждому критерию
- Полные правила: [[Регламент_Субагента]]

### Ты — оркестратор
Если ты управляешь другими агентами:
- **Сначала планировщик** — всегда, без исключений
- Не пишешь код и не редактируешь файлы сам — только управляющий контур
- Проверяешь план → запускаешь субагентов → проверяешь доказательства → интегрируешь
- Полные правила: [[Регламент_Оркестратора]]
- Краткий промпт: [[Промпт_Правил_Оркестратора]]

### Режимы работы (для субагентов)
- **M0** — локальная задача: один владелец, минимум координации
- **M1** — Discovery/Lean MVP: быстрая проверка гипотезы
- **M2** — Multi-Stream: несколько потоков, нужна интеграция
- **M3** — Hardening/Release: риски релиза, безопасность, откат
