# ADR: Модульный монолит

## Статус

Принято

## Контекст

Проект Focus — планировщик задач с бэкендом на Spring Boot, web SPA на React и Android-клиентом. На старте команда из одного разработчика, требования к масштабированию минимальны. Необходимо выбрать архитектурный стиль бэкенда.

## Альтернативы

1. **Микросервисы** — отдельные сервисы на каждый модуль, независимый деплой
2. **Модульный монолит** — единый деплой-артефакт, разделение на модули внутри
3. **Монолит без модулей** — единый jar без внутренних границ

## Решение

Выбран **модульный монолит**: один Spring Boot jar, одна база данных, внутреннее разделение на модули (`auth`, `tasks`, `folders`, `recurrence`, `reminders`, `prioritypolicy`, `notifications` и др.). Каждый модуль — отдельный package со своим API (интерфейсы), изоляция на уровне компиляции.

## Обоснование

- **Простота разработки:** один репозиторий, один CI/CD пайплайн, один деплой
- **Достаточная изоляция:** модули зависят только от API-интерфейсов друг друга, а не от реализаций
- **Лёгкий рефакторинг в микросервисы:** при необходимости модуль извлекается в отдельный сервис заменой in-process вызовов на REST/gRPC
- **Низкий DevOps overhead:** не нужен service mesh, распределённые транзакции, сложный мониторинг
- **Соответствие стадии:** на этапе Init/MVP микросервисы — избыточное усложнение

## Последствия

- Необходима дисциплина: модули не должны иметь циклических зависимостей
- Тесты должны проходить на уровне модуля без поднятия всего контекста
- При росте команды/нагрузки потребуется план миграции на микросервисы (но не сейчас)
