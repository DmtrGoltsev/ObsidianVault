# ADR: JWT-аутентификация

## Статус

Принято

## Контекст

Focus имеет три клиента: Web SPA (React), Android (Kotlin) и потенциально сторонние интеграции. Необходим stateless-механизм аутентификации, не требующий серверных сессий.

## Альтернативы

1. **Session-based (cookie + Redis)** — серверные сессии, хранение в Redis
2. **JWT (access + refresh)** — токены, клиент хранит refresh
3. **OAuth2/OIDC** — делегирование внешнему провайдеру (Google, GitHub)

## Решение

Выбрана **JWT-аутентификация** с двумя токенами:

- **Access token:** short-lived (15 минут), содержит `userId`, `roles`. Используется для авторизации запросов.
- **Refresh token:** long-lived (30 дней), хранится в httpOnly cookie (Web) или защищённом хранилище (Android). Используется для получения нового access-токена.

## Обоснование

- **Stateless:** бэкенд не хранит сессии, не требует Redis
- **Cross-platform:** работает одинаково на Web и Android
- **Безопасность:** короткий срок access-токена ограничивает окно компрометации
- **Refresh token:** позволяет «тихий» релог без ввода пароля
- **Простота:** Spring Security + `jjwt` библиотека, минимальный код
- **OAuth2 избыточен:** на старте нет необходимости в social login

## Последствия

- Клиенты должны реализовать логику обновления токенов (interceptor на 401)
- Необходим механизм отзыва refresh-токенов (blacklist в БД)
- Access-токен не должен содержать конфиденциальных данных (подписан, но не зашифрован)
- При компрометации refresh-токена злоумышленник получает доступ на 30 дней
