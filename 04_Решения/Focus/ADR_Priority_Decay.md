# ADR: Раздельный Priority Decay для зелёных и красных задач

## Статус

Принято

## Контекст

В системе Focus есть два типа приоритетных задач: зелёные ([[Green_Task]] — осознанно выбранные пользователем) и красные ([[Red_Task]] — автоматический индикатор прокрастинации). Требуется определить политику автоснижения приоритета ([[Priority_Decay]]) для каждого типа.

## Альтернативы

1. **Единый коэффициент decay** — все задачи теряют приоритет одинаково
2. **Раздельные коэффициенты** — зелёные и красные задачи имеют разную скорость decay
3. **Только decay для красных** — зелёные задачи не теряют приоритет

## Решение

Выбраны **раздельные коэффициенты decay**:

- `green_decay_factor` — медленное снижение (например, 0.95 в час). Зелёная задача важна осознанно, пользователь не должен терять её из виду быстро.
- `red_decay_factor` — быстрое снижение (например, 0.85 в час). Красная задача — индикатор прокрастинации, быстрое падение стимулирует пользователя к действию.

## Обоснование

- **Разная семантика:** зелёный — осознанный выбор, красный — автоматический стимул. Разное поведение decay отражает эту разницу.
- **Пользовательский опыт:** если зелёные задачи теряют приоритет так же быстро, как красные, пользователь теряет мотивацию вручную помечать важное.
- **Геймификация:** красный decay создаёт urgency, зелёный decay мягко напоминает о важном.
- **Простота реализации:** два коэффициента вместо одного — минимальное усложнение кода.

## Последствия

- В БД необходимо хранить поле `task_type` (GREEN/RED) или отдельный флаг
- Планировщик должен применять правильный коэффициент в зависимости от типа
- При ручном снятии зелёного флага задача переходит на обычный (или красный) decay
- UI должен визуально различать скорость decay (индикатор, цвет)
