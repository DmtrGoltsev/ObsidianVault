# ADR: JWT access + refresh токены

## Контекст

Требовалось выбрать механизм аутентификации и авторизации для REST API, обслуживающего веб-клиент и Android-клиент.

## Решение

Использовать [[JWT]] (JSON Web Token) с парой access + refresh токенов.

## Обоснование

- **Stateless** — бэкенд не хранит сессии, что упрощает горизонтальное масштабирование (если потребуется в будущем)
- **Access токен** — короткоживущий (минуты), передаётся в Authorization header
- **Refresh токен** — долгоживущий (дни), хранится клиентом, используется для обновления access токена без повторного ввода пароля
- Spring Security + JWT — зрелая связка с богатой экосистемой
- Подходит для обоих клиентов (веб, Android) без изменений протокола

## Последствия

- Токены необходимо безопасно хранить на клиенте (HttpOnly cookie для веб, EncryptedSharedPreferences для Android)
- Компрометация refresh токена требует механизма отзыва (blacklist)
- Все запросы к API требуют валидный access токен

## Альтернативы

- **Session-based auth (JSESSIONID)** — отвергнут: требует sticky sessions, хуже для mobile-клиентов
- **OAuth2/OIDC** — отвергнут: избыточно для B2C приложения без федеративной аутентификации
