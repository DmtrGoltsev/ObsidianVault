# ADR: Модульный монолит

## Контекст

При старте проекта стоял выбор архитектурного стиля для бэкенда: микросервисы или модульный монолит. Проект — B2C приложение планирования задач с ограниченной командой.

## Решение

Выбран [[Modular_Monolith|модульный монолит]] на Spring Boot вместо микросервисной архитектуры.

## Обоснование

- Малая команда — микросервисы создали бы избыточную операционную сложность
- Нет требований к независимому масштабированию модулей
- Модульный монолит позволяет выделить модули (auth, accounts, folders, goals, tasks, sharing, calendar, recurrence, reminders, prioritypolicy, notifications) с чёткими границами, но в едином процессе и БД
- Миграция к микросервисам в будущем упрощена за счёт уже выделенных модульных границ

## Последствия

- Единая кодовая база, единый деплой
- Общая [[PostgreSQL_Advisory_Lock|PostgreSQL]] БД
- [[Flyway]] управляет схемой централизованно
- CI/CD проще: один Docker-образ, один пайплайн

## Альтернативы

- **Микросервисы** — отвергнуты: избыточно для текущего масштаба команды и нагрузки
- **Монолит без модулей** — отвергнут: усложнил бы навигацию и будущую декомпозицию
