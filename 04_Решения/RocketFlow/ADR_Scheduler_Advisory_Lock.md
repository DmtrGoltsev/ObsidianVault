# ADR: PostgreSQL advisory lock для планировщика напоминаний

## Контекст

Планировщик напоминаний периодически сканирует задачи с истекающими reminder-правилами и отправляет push-уведомления. При запуске нескольких экземпляров бэкенда возможна двойная отправка одного напоминания.

## Решение

Использовать [[PostgreSQL_Advisory_Lock]] для синхронизации единственного активного планировщика.

## Обоснование

- Advisory lock на уровне PostgreSQL гарантирует, что только один экземпляр бэкенда выполняет цикл планировщика в данный момент
- В отличие от блокировок на уровне приложения (Redis, etc.), не требует дополнительной инфраструктуры
- Автоматически освобождается при падении экземпляра (сессия БД разрывается — lock снимается)
- Простая реализация: `pg_try_advisory_lock(lock_id)` перед запуском цикла

## Последствия

- Все экземпляры бэкенда должны подключаться к той же БД (что так и есть в [[ADR_Модульный_Монолит|модульном монолите]])
- Если экземпляр с блокировкой зависает (но не падает), блокировка держится до таймаута — требуется мониторинг
- Не масштабирует обработку напоминаний (однопоточная модель)

## Альтернативы

- **Redis-блокировка (Redlock)** — отвергнута: требует Redis в инфраструктуре
- **Без синхронизации** — отвергнуто: двойная отправка уведомлений недопустима
