# ADR: Оптимистичная блокировка

## Контекст

Несколько пользователей могут одновременно редактировать одну задачу при совместном доступе. Требовался механизм предотвращения конфликтов конкурентных обновлений.

## Решение

Использовать [[Optimistic_Locking|оптимистичную блокировку]] через поле `version` в JPA-сущностях.

## Обоснование

- Совместное редактирование одной задачи — редкий случай (не hot path)
- Пессимистичная блокировка излишне снижает производительность (блокировки строк на всё время транзакции)
- JPA `@Version` — нативная поддержка в Spring Data, минимальный код
- При конфликте версий выбрасывается `OptimisticLockException`, клиент получает 409 Conflict и может перечитать актуальные данные
- Не требует координации между экземплярами приложения (stateless)

## Последствия

- Клиент должен обрабатывать 409 Conflict и реализовывать повтор с обновлённой версией
- Не подходит для сценариев с высокой частотой конкурентных правок (что не является случаем RocketFlow)

## Альтернативы

- **Пессимистичная блокировка (SELECT FOR UPDATE)** — отвергнута: блокирует строки, снижает пропускную способность
- **Last-write-wins** — отвергнут: потеря данных при конкурентных правках
