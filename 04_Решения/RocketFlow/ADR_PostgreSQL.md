# ADR: PostgreSQL как единая БД

## Контекст

Требовалось выбрать СУБД для хранения всех данных приложения: пользователи, папки, цели, задачи, напоминания, устройства.

## Решение

Использовать PostgreSQL 16 как единственную реляционную базу данных.

## Обоснование

- Зрелая, надёжная СУБД с открытым исходным кодом
- Отличная поддержка в Spring Data JPA и [[Flyway]]
- Встроенные механизмы конкурентного доступа: [[Optimistic_Locking|оптимистичная блокировка]], [[PostgreSQL_Advisory_Lock|advisory locks]]
- Embedded PostgreSQL (zonky) для тестов — изоляция CI от внешних зависимостей
- Нет необходимости в NoSQL: данные строго реляционные (связи папка→цель→задача, пользователи→устройства, авторизация)

## Последствия

- Единая схема — [[Flyway]] миграции (18: V1–V18)
- [[Soft_Delete|Мягкое удаление]] через SQL-фильтры
- Advisory lock для планировщика напоминаний гарантирует однопоточную обработку

## Альтернативы

- **MySQL/MariaDB** — отвергнуты: слабее поддержка advisory locks, хуже интеграция с zonky embedded
- **NoSQL (MongoDB)** — отвергнут: данные реляционные, не подходит
