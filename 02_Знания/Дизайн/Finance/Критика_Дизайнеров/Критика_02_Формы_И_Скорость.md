## Критические наблюдения

### P0. Android OCR privacy narrative расходится с фактическим UI path

**Наблюдение.** Product docs и user guide говорят: Android OCR выполняется локально/on-device без загрузки скриншота. В Android UI `processScreenshotCapture` читает bytes выбранного изображения и вызывает `apiClient.screenshotOcr(...)`; текст статуса: "Отправляем скриншот на распознавание".

**Доказательства / пути.** `docs/product-mvp.md`, `docs/user-guide-ru.md`, `docs/current-status.md`; `apps/android/.../FinanceApp.kt` около `processScreenshotCapture`; journey map уже помечает implementation mismatch как открытый риск.

**Почему это forms/friction.** OCR — ускоритель ручного ввода. Если пользователь верит, что скрин не уходит с устройства, а UI фактически отправляет его на backend, это не просто friction: это нарушение доверия к самому shortcut-сценарию.

**Рекомендация.** До публичной UX-копии выбрать один truth source: либо реально on-device, либо честная временная backend-обработка с retention/privacy пояснением. В форме OCR рядом с upload-кнопкой показывать короткий privacy state: "обрабатывается на устройстве" или "временно отправляется на self-hosted backend, raw не сохраняется".

### P1. Android Quick Add позволяет выбрать `Обзор` как visibility для создания записи

**Наблюдение.** `QuickAddSheet` строит chips из `FinanceMode.entries`, а enum содержит `Personal`, `Shared`, `Overview`. Для submit логика фактически трактует не-Shared как personal. Пользователь может выбрать "Обзор" в форме, хотя обзор — режим чтения, не scope записи.

**Доказательства / пути.** `FinanceApp.kt`: `FinanceMode` содержит `Overview`; `QuickAddSheet` использует `ChipRow(FinanceMode.entries.toList(), visibility, ...)`; submit для asset вычисляет ownership как shared только при `FinanceMode.Shared`, иначе personal.

**Почему это forms/friction.** "Обзор" в форме создания — ложный выбор. Пользователь думает, что добавляет в агрегированный режим, а запись уйдет в personal. Это риск неверного учета и недоверия к семейной модели.

**Рекомендация.** В формах создания оставить только `Личное` и `Общее`; если текущий экран в Overview, default должен быть явным выбором, не автоперсонал. Для Overview показывать compact prompt: "Выберите, куда записать: личное или общее".

### P1. PWA Quick Add по умолчанию подставляет сумму `1000`

**Наблюдение.** PWA `QuickAdd` открывается с `amount = "1000"`. Это ускоряет демо, но в реальном ежедневном учете опасно: пользователь может сохранить не ту сумму, особенно на мобильном.

**Доказательства / пути.** `apps/web-pwa/src/App.tsx`, `function QuickAdd`, начальный state `amount`.

**Почему это forms/friction.** В финансовом продукте пустая сумма лучше, чем уверенный неправильный default. Скорость не должна достигаться предзаполнением денежного факта.

**Рекомендация.** Сделать сумму пустой, открыть цифровую клавиатуру, сохранить последний выбранный тип/счет/категорию, но не сумму. Для повторяющихся платежей нужен отдельный pattern "повторить прошлую операцию", а не глобальный default.

### P1. PWA transfer form не объясняет same-scope / same-currency ограничения до submit

**Наблюдение.** PWA `canSubmit` для transfer проверяет только наличие двух разных account ids. Same-scope, household compatibility и currency rules не отражены в UI; backend или API client должны ловить это после попытки.

**Доказательства / пути.** `apps/web-pwa/src/App.tsx`, `QuickAdd canSubmit`; `README.md` фиксирует transfers same-scope only; Android имеет `transferPairValidationMessage`, а PWA аналогичного явного preflight не показывает.

**Почему это forms/friction.** Перевод — частый сценарий: карта -> вклад, карта -> наличные. Ошибка после submit ощущается как поломка, а не как понятное правило приватности/валюты.

**Рекомендация.** На выборе "Куда" фильтровать destination accounts до совместимых; под полем показывать короткую причину, если вариантов нет: "Для перевода нужны два счета одного режима и валюты".

### P1. PWA transfer использует `allAccounts`, а не видимые accounts текущего режима

**Наблюдение.** В PWA Quick Add передаются `accounts={visibleAccounts(...)}` и `allAccounts={snapshot.accounts.filter(status !== deleted)}`. Для transfer picker используются `allAccounts`, поэтому пользователь в personal/shared контексте может видеть варианты шире текущего режима.

**Доказательства / пути.** `apps/web-pwa/src/App.tsx`: props `accounts`/`allAccounts`; transfer source/destination selects map over `allAccounts`.

**Почему это forms/friction.** Контекст режима сверху говорит одно, форма выбора счетов показывает другое. Это увеличивает когнитивную проверку и повышает риск ошибки scope.

**Рекомендация.** Разделить два сценария: обычная операция использует счета текущего режима; transfer может иметь "показать совместимые счета" с явным scope badge на каждом счете. Не показывать несовместимые варианты как равноправные.

### P1. Исторически подтвержденный UX FAIL: invalid transfer pair не показывал понятное сообщение

**Наблюдение.** Android Re-QA 2026-05-18 фиксировал FAIL: после сохранения personal -> shared transfer sheet оставался без видимого error/snackbar/dialog. Сейчас код содержит `quickAddError` и `quick-add-error`, но это место требует regression lock, потому что риск уже материализовался.

**Доказательства / пути.** `MVP_EVIDENCE/.../android-device-rerun-qa-2026-05-18.md`; `FinanceApp.kt` `quickAddError`; `AppSectionTest.kt` проверяет тексты `transferPairValidationMessage`.

**Почему это forms/friction.** Невидимая ошибка — худший тип friction: пользователь уже принял решение и не понимает, что исправить.

**Рекомендация.** Зафиксировать обязательную визуальную проверку на Android и PWA: invalid pair показывает inline error в sheet, фокус/scroll подводит к ошибке, submit остается доступным только если есть actionable reason или preflight.

### P1. Android Quick Add auto-creates missing account/category, что ускоряет демо, но размывает контроль

**Наблюдение.** Android submit для expense/income при отсутствии account/category создает demo account/category. Для живого финансового учета это может породить "автоматические" сущности с неясными названиями и последующей чисткой.

**Доказательства / пути.** `FinanceApp.kt` `submitQuickAdd`: `createDemoAccount`, `createDemoCategory`, `createDemoTransaction`; QA fixes содержит работу по seed/demo labels.

**Почему это forms/friction.** Быстрее сейчас, дороже завтра: пользователь получает операцию, но потом вынужден разбирать категории/счета, которые он явно не создавал.

**Рекомендация.** Для production UX заменить auto-create на embedded create mini-flow: "Нет категории расходов — создать за 1 поле" с preview имени/scope. Auto-create оставить только в dev seed/demo режиме.

### P1. Категории в PWA расхода/дохода можно оставить `Без категории`, что ухудшает аналитику

**Наблюдение.** PWA Quick Add предлагает `<option value="">Без категории</option>` и может отправить `categoryId: null`. Это полезно как escape hatch, но ежедневный учет потом теряет смысловую структуру.

**Доказательства / пути.** `apps/web-pwa/src/App.tsx`, `QuickAdd` category select и `createDemoOperation(categoryId: input.categoryId || null)`; product docs говорят, что категории помогают читать аналитику.

**Почему это forms/friction.** Пользователь экономит секунду при вводе, но платит минутами в аналитике. Особенно плохо для top categories и planning факта.

**Рекомендация.** Оставить "Без категории" только как осознанное вторичное действие: default — последняя/самая вероятная категория; если пусто, показать "Создать категорию" и "Продолжить без категории".

### P1. Planning allocation editor перегружен: target type, target, recurrence, amount/percent, savings goal, due month, comment в одном блоке

**Наблюдение.** `PlanningAllocationEditor` содержит много решений в одном непрерывном editor. Для power user это работает; для первого планирования это выглядит как конструктор правил, а не быстрый бюджет.

**Доказательства / пути.** `PlanningUi.kt`: `PlanningAllocationEditor`, `AllocationsCard`; journey map выделяет planning allocation/savings goal как критичный путь.

**Почему это forms/friction.** Planning — не ежедневный ввод, но если форма тяжелая, пользователь не создаст план, и daily accounting потеряет ориентир "план/факт".

**Рекомендация.** Разделить на fast path и advanced: сначала "категория/цель + сумма", затем collapsible "повторяемость, процент, накопительная цель, комментарий". Savings goal показывать как отдельный mode после выбора investment asset category.

### P1. Disabled states в planning/Quick Add часто не называют причину рядом с кнопкой

**Наблюдение.** В Android и PWA есть кнопки `enabled = ...` / `disabled={!canSubmit}`, но не всегда есть текст причины рядом с кнопкой: пустая сумма, нет target, нет категории, invalid goal amount, нет household.

**Доказательства / пути.** `PlanningUi.kt`: `canCreateAllocation`, save buttons, create sheets; `App.tsx`: PWA `QuickAdd canSubmit`, OCR `canConfirm`, category `canSubmit`; journey map отдельно требует показывать причины disabled submit.

**Почему это forms/friction.** Disabled без объяснения заставляет пользователя угадывать. В финансовых формах угадывание быстро превращается в отказ от учета.

**Рекомендация.** Ввести единый паттерн `formBlockingReason`: одна короткая строка над primary button. Примеры: "Введите сумму", "Выберите категорию", "Для общего режима нужен household", "Цель накопления требует сумму и срок".

### P2. Empty states слишком общие и не ведут к следующему действию

**Наблюдение.** PWA empty states: "Нет активов в этом режиме", "Пока нет расходов", "Нет категорий в этом режиме", "Нет черновиков для подтверждения". Android: "Операций пока нет", "Активов пока нет", "Распределений пока нет". Они описывают пустоту, но не сокращают путь.

**Доказательства / пути.** `apps/web-pwa/src/App.tsx` `EmptyState`; `FinanceApp.kt` empty texts; `PlanningUi.kt` empty texts.

**Почему это forms/friction.** Empty state — шанс начать учет. Сейчас он скорее сообщает, что данных нет, чем ведет к "создать первый счет" / "добавить расход" / "создать категорию".

**Рекомендация.** Каждый empty state должен иметь next-best-action с учетом scope: "Создать личный счет", "Создать общую категорию", "Добавить расход вручную", "Выбрать скрин расходов", "Создать план".

### P2. Operations entry point конкурирует с OCR и ручным вводом без ясной иерархии

**Наблюдение.** В Operations одновременно живут capture drafts, screenshot OCR и обычный timeline; глобальная кнопка "Добавить" открывает Quick Add. Пользователь может не понять, что быстрее для обычной покупки: Quick Add или скрин.

**Доказательства / пути.** `FinanceApp.kt` `operationsContent` сначала показывает `CaptureDraftReviewCard`; `App.tsx` `OperationsPage` включает OCR и pending drafts; journey map выделяет OCR как assistant, не import.

**Почему это forms/friction.** Если у пользователя нет скрина, он должен за секунду попасть в ручной расход. OCR не должен заслонять основной ежедневный маршрут.

**Рекомендация.** В Operations сделать явную первичную кнопку "Расход" / "Доход" / "Перевод"; OCR держать как secondary action "Из скрина". Pending drafts поднимать наверх только когда они реально есть.

### P2. Термины "счет", "актив", "категория активов", "цель" смешиваются в формах

**Наблюдение.** Product docs разделяют деньги/операции/активы/категории, но UI местами называет счет "активом", а planning target — "целью". Для investment asset category это особенно чувствительно, потому что account target больше не должен возвращаться для новых allocations.

**Доказательства / пути.** `MOC_Finance` и `QA_Фиксы` фиксируют asset categories as source of truth; `PlanningUi.kt` target labels; `AssetsPage` / `AssetTile`; journey map предупреждает не возвращать account target.

**Почему это forms/friction.** Пользователь выбирает не техническую сущность, а смысл: "куда планирую", "откуда списать", "где лежат деньги". Непоследовательные термины тормозят выбор.

**Рекомендация.** В формах использовать стабильные labels: "Счет списания", "Счет получения", "Категория расхода", "Инвестиционная категория", "Цель накопления". Не называть все подряд "актив".

### P2. Нет явного "repeat last" / шаблонов для ежедневного учета

**Наблюдение.** Текущий Quick Add ускоряет ввод через компактную форму, но не использует самый сильный паттерн личных финансов: повторить последнюю похожую операцию.

**Доказательства / пути.** В UI code видны Quick Add, timeline и category/account pickers; отдельного repeat pattern в изученных Android/PWA UI фрагментах не обнаружено.

**Почему это forms/friction.** Ежедневный учет — это десятки похожих операций: кофе, продукты, транспорт, перевод на вклад. Повторение снижает количество полей с 4-6 до 1-2.

**Рекомендация.** Добавить quick suggestions из последних операций: "Повторить продукты", "Снова транспорт", "Перевод на вклад". При tap открывать форму с заполненными счетом/категорией/типом, но пустой или редактируемой суммой.

## Рекомендации по упрощению

1. Сделать две формы ежедневного ввода: ultra-fast `Расход/Доход/Перевод` и expanded details. В ultra-fast оставить: сумма, счет, категория/получатель. Дата today и комментарий свернуты.
2. Убрать `Обзор` из всех creation scopes. Overview может быть экраном чтения и entry point, но не значением записи.
3. Ввести единый блокирующий текст для disabled primary button во всех формах.
4. Для transfer фильтровать несовместимые счета до выбора, а не после submit.
5. Для пустых счетов/категорий давать inline create mini-flow, не auto-create demo entity.
6. В Planning вынести advanced options под раскрытие; default allocation = target + amount.
7. OCR позиционировать как secondary accelerator: "Из скрина", а не как главный путь операций.
8. Добавить repeat-last suggestions без автоподстановки суммы по умолчанию.
9. Везде показывать scope badge на сущностях, которые выбираются в форме.
10. Синхронизировать Android/PWA copy: одинаковые названия режимов, transfer rules, OCR privacy state.

## Что НЕ менять

- Не превращать MVP в банковский импорт, SMS/push interception или bank API. Это явно вне scope и требует отдельного security/privacy review.
- Не возвращать account target для новых planning allocations; текущая граница — expense category или investment asset category.
- Не вводить plan-level progress bar: release evidence фиксирует progress на allocation-level.
- Не скрывать перевод внутри расходов: transfer должен оставаться отдельным типом и не портить expense analytics.
- Не усложнять personal/shared privacy модель ради "удобного общего обзора". Overview не равен всем personal данным семьи.
- Не делать OCR автосозданием операции; draft должен оставаться pending до подтверждения пользователя.
- Не добавлять инвестиционные рекомендации, доходность, котировки или налоговую аналитику в формы активов/планирования.

## Критерии успеха

- Пользователь может добавить типичный расход за 10-15 секунд: открыть Quick Add, ввести сумму, подтвердить счет/категорию, сохранить.
- В форме создания нет значения `Обзор`; каждая новая запись явно personal или shared.
- 100% disabled primary buttons в Quick Add, OCR confirm, CategoryForm, Planning allocation/source имеют видимую причину.
- Transfer destination picker показывает только совместимые счета или объясняет, почему совместимых счетов нет.
- Empty states ведут к одному следующему действию, а не только сообщают "нет данных".
- OCR copy на Android/PWA совпадает с фактической архитектурой обработки изображения.
- Planning allocation можно создать через fast path из двух решений: target и сумма; savings goal не мешает обычному распределению.
- Regression evidence включает Android и PWA screenshots для invalid transfer, no accounts, no categories, shared without household, OCR no results, planning no targets.

## Риски

- Privacy risk: неверная OCR-копия может создать ложное ощущение on-device обработки.
- Data quality risk: `Без категории`, demo auto-create и wrong default amount ухудшают аналитику сильнее, чем помогают скорости.
- Scope risk: `Обзор` в формах и allAccounts в transfer picker могут привести к ошибочным ожиданиям personal/shared поведения.
- Trust risk: invisible validation или молчаливые disabled states создают ощущение нестабильного приложения.
- Regression risk: fixes подтверждены сборкой и частично runtime smoke, но не все формы имеют актуальную exhaustive manual QA matrix.
- Product creep risk: попытка "ускорить ввод" через импорт/SMS/API уведет за границы MVP и security gates.
