# Optimistic Locking (Оптимистичная блокировка)

## Определение

Стратегия конкурентного доступа, при которой конфликты обнаруживаются в момент сохранения, а не предотвращаются заранее. Используется в RocketFlow через Spring Data JPA (аннотация `@Version`).

## Контекст в проекте

- Применяется к доменным сущностям, которые могут редактироваться одновременно (например, [[Task|задачи]] при совместном доступе через [[Collaborator|соавторов]]) (предположение)
- При конфликте выбрасывается исключение, клиент должен повторить операцию (предположение)
- Дополняет [[PostgreSQL_Advisory_Lock]] для сценариев с низкой конкуренцией

## Бизнес-правила

- Поле `version` в сущностях JPA
- Конфликт = ошибка сохранения, не молчаливая потеря данных

## Связанные термины

- [[PostgreSQL_Advisory_Lock]] — дополнительный механизм для высококонкурентных сценариев
- [[Soft_Delete]] — другой persistence-паттерн
- [[Task]] — сущность, к которой применяется
